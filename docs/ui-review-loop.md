# UI 수정 → 디자인 리뷰 → Gemini/Claude 합의 루프

Keepgoing-FE에서 페이지/UI를 개선할 때 사용할 수 있는 반복 작업 플레이북이다.

## 목적

다음 루프를 반복해, 기능과 디자인 모두에서 더 이상 액션 가능한 피드백이 없을 때 작업을 종료한다.

1. 페이지 수정
2. 내부 디자인 리뷰
3. 부족한 점 수정
4. Gemini / Claude 외부 리뷰
5. 외부 피드백 반영
6. 무피드백 상태가 되면 완료

---

## 역할

### 1) 메인 구현 에이전트
- 실제 코드 수정 담당
- 한 번에 하나의 명확한 개선 목표만 처리
- 매 반복마다 변경 이유와 검증 결과를 남김

### 2) 내부 디자인 리뷰어
- `designer`, `vision`, 필요 시 `visual-verdict`
- 아래 항목 위주로 평가
  - 레이아웃 균형
  - 시각적 위계
  - 여백/밀도
  - 타이포그래피
  - 버튼/입력창/카드의 일관성
  - 초심자 입장에서의 이해 가능성

### 3) 외부 리뷰어
- `ask-gemini`
- `ask-claude`
- 내부 리뷰 통과 뒤 마지막 편향 제거용으로 사용
- “좋다/별로다”보다 **실행 가능한 개선점**만 수집

---

## 종료 조건

아래를 모두 만족하면 종료한다.

1. `lint`, `test`, 필요 시 `e2e`가 통과한다.
2. 내부 디자인 리뷰에서 **blocker / major 이슈 0개**.
3. Gemini와 Claude 모두 **액션 가능한 신규 피드백 0개**.
4. 현재 변경사항에 대해 알려진 기능 회귀가 없다.

---

## 반복 규칙

- 기본 최대 반복 횟수: **3~5회**
- 한 라운드에는 **한 가지 핵심 문제만** 해결한다.
  - 예: “정보 위계 개선”, “밀도 완화”, “CTA 강조”, “폼 가독성 향상”
- 같은 피드백이 2회 이상 반복되면 구조적 문제로 간주하고 더 큰 수정으로 전환한다.
- Nit 수준만 남으면 종료 가능하다.

---

## 표준 루프

## 0. 시작 전 기준선 확보

### 실행
```bash
npm run lint
npm run test:run
```

필요하면:
```bash
npm run test:e2e
```

### 확보할 것
- 현재 화면 스크린샷
- 작업 대상 페이지 경로
- 이번 라운드의 목표 1개

예시 목표:
- 대시보드 첫 화면의 시선 흐름을 단순화한다.
- 노트 리스트의 밀도를 낮추고 초심자 이해도를 높인다.
- 메인 CTA와 보조 액션의 위계를 분명하게 만든다.

---

## 1. 페이지 수정

### 원칙
- 작은 diff 우선
- 기존 패턴 재사용
- 새 의존성 추가 금지
- 스타일만 바꾸지 말고, 정보 구조도 같이 본다

### 수정 시 체크리스트
- 첫 화면 3초 안에 무엇을 해야 하는지 보이는가?
- 가장 중요한 액션이 가장 잘 보이는가?
- 비슷한 컴포넌트가 비슷하게 보이는가?
- 정보량이 많아도 숨이 막히지 않는가?
- 색이 의미 없이 많지 않은가?

---

## 2. 내부 디자인 리뷰

### 리뷰 입력물
- 현재 브랜치 화면 스크린샷
- 작업 목표
- 바뀐 포인트 요약 3줄 이내

### 리뷰 질문 템플릿
```text
이 화면을 비주얼/UX 관점에서 리뷰해줘.

목표:
- {이번 라운드 목표}

중점 평가:
- 정보 위계
- 여백과 밀도
- 타이포 가독성
- CTA 명확성
- 초심자 친화성

출력 형식:
1. blocker
2. major
3. minor
4. nit
5. 바로 수정할 1~3개 액션
```

### 내부 리뷰 통과 기준
- blocker 0
- major 0 또는 1개 이하
- 수정 액션이 구체적일 것

`visual-verdict`를 쓸 경우 목표 기준:
- `score >= 90`
- `verdict = pass`

---

## 3. 내부 피드백 반영

### 반영 방식
- 가장 영향이 큰 1~3개만 반영
- 한 번에 너무 많이 고치지 않기
- 반영 후 다시 스크린샷 생성

### 이 단계에서 반드시 할 것
```bash
npm run lint
npm run test:run
```

수정 영향이 크면:
```bash
npm run test:e2e
```

---

## 4. Gemini / Claude 외부 리뷰

내부 리뷰에서 더 이상 큰 이슈가 없을 때만 실행한다.

### 공통 원칙
- 둘 다 같은 스크린샷, 같은 목표, 같은 질문으로 물어본다.
- “취향”보다 **실행 가능한 비판**만 받는다.
- 이미 해결한 이슈를 다시 끌어오지 않도록 현재 의도를 설명한다.

### 프롬프트 템플릿
```text
React/Vite 기반 Keepgoing-FE 화면 리뷰를 부탁해.

대상:
- {페이지 이름}

이번 작업 목표:
- {이번 라운드 목표}

현재 의도:
- {이번 수정의 핵심 의도 2~4줄}

리뷰 기준:
- 정보 위계
- 화면 첫인상
- CTA 명확성
- 복잡도/밀도
- 초심자 친화성
- 시각적 일관성

중요:
- 단순 취향 평가는 제외
- 실제로 고칠 가치가 있는 점만 말해줘
- 중복 피드백은 제외
- 결과는 아래 형식으로 정리해줘

출력 형식:
1. 반드시 고쳐야 하는 점
2. 고치면 좋아지는 점
3. 그대로 둬도 되는 점
4. 결론: 추가 수정 필요 여부 (yes/no)
```

### Gemini 호출 규칙
로컬 이미지가 있으면 절대경로를 프롬프트에 inline으로 넣는다.

예시:
```bash
gemini -p "이 이미지를 기준으로 Keepgoing-FE 대시보드를 리뷰해줘: @/absolute/path/to/screenshot.png"
```

또는 skill 경로:
```bash
omx ask gemini "{프롬프트 내용 + @/absolute/path/to/screenshot.png}"
```

### Claude 호출 예시
```bash
omx ask claude "{프롬프트 내용}"
```

### 산출물 저장 위치
- `.omx/artifacts/gemini-*.md`
- `.omx/artifacts/claude-*.md`

---

## 5. 외부 피드백 합의 정리

Gemini/Claude 결과를 아래 규칙으로 통합한다.

### 즉시 반영
- 둘 다 지적한 문제
- 사용성/위계/가독성 관련 문제
- 구현 비용이 낮고 효과가 큰 문제

### 보류 가능
- 취향 차이만 있는 문제
- 현재 목표와 무관한 문제
- 구조 전체를 다시 짜야 하는데 근거가 약한 문제

### 무시 가능
- 이미 해결된 피드백
- 실제 화면과 맞지 않는 오해
- 제품 방향과 충돌하는 제안

합의 결과는 아래 형식으로 정리한다.

```text
[반영]
- ...

[보류]
- ...

[무시]
- ...

[다음 라운드 핵심 목표]
- ...
```

## 6. 최종 검증

작업 종료 직전 최소 검증:

```bash
npm run lint
npm run test:run
npm run test:e2e
```

추가로 남길 것:
- 최종 스크린샷
- 최종 변경 파일 목록
- 이번 라운드에서 반영한 핵심 개선점
- 남은 minor/nit 리스크

## 권장 실행 순서 예시

```text
라운드 1
- 목표 설정
- 페이지 수정
- lint/test
- 내부 디자인 리뷰
- major 있으면 재수정

라운드 2
- 재수정본 캡처
- 내부 리뷰 재실행
- 통과 시 Gemini/Claude 리뷰
- 공통 피드백만 반영

라운드 3
- 최종 미세 조정
- lint/test/e2e
- 추가 수정 필요 없음 확인
- 완료
```

## 빠른 운영 체크리스트

### 매 라운드 시작
- [ ] 이번 라운드 목표 1개를 적었다.
- [ ] 현재 화면 스크린샷을 확보했다.

### 수정 후
- [ ] lint 통과
- [ ] unit/integration test 통과
- [ ] 필요한 경우 e2e 통과
- [ ] 스크린샷 갱신

### 내부 리뷰 후
- [ ] blocker 0
- [ ] major 0 또는 수용 가능한 수준
- [ ] 다음 액션 1~3개가 명확함

### 외부 리뷰 후
- [ ] Gemini 결과 확인
- [ ] Claude 결과 확인
- [ ] 공통 피드백만 추출
- [ ] 취향성 피드백은 분리

### 종료 전
- [ ] 액션 가능한 신규 피드백 0개
- [ ] 기능 회귀 없음
- [ ] 최종 검증 완료

## 이 프로젝트에서의 권장 적용 대상

특히 아래 파일/화면 계열에 잘 맞는다.

- `src/pages/*`
- 대시보드 / 노트 리스트 / 블로그 리스트 같은 정보 밀도 높은 화면
- 첫인상과 사용성 개선이 중요한 랜딩성 UI

---

## 실무 운영 팁

- 디자인 리뷰는 “예쁜가?”보다 “사용자가 덜 헷갈리는가?”를 우선한다.
- Gemini/Claude는 최종 판정자가 아니라 **편향 제거용 외부 리뷰어**로 쓴다.
- 둘 다 조용하고 내부 리뷰도 조용하면 종료한다.
- 끝없이 다듬지 않도록 최대 반복 횟수를 넘기면 마감 판단을 한다.
